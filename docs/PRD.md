# Delicious 产品需求文档 (PRD)

> 版本: v2 (2026-06-30 修订)
> 状态: V1.0 设计冻结，进入开发

## 1. 产品定位

一款 AI 驱动的"美食日记 + 厨房助手" App。用户拍一张菜的照片，AI 自动识别菜名、生成材料清单、估算成本、写出制作步骤。记录沉淀为个人菜单，可随时复用、生成购物清单、查看烹饪开销。

**一句话**：拍照即所得的美食日记，让"我做过/我想再做"的菜不再丢失。

## 2. 目标用户与场景

| 用户画像 | 高频场景 | 我们解决的痛点 |
|---|---|---|
| 周末家庭主厨 | 周日做了一桌菜，想留个记录 | 拍照即记录，不用手动写菜谱 |
| 想学做饭的新手 | 看到外卖里的菜想自己做 | AI 给出可执行步骤和购物清单 |
| 精打细算的家庭烹饪者 | 月底想看伙食花了多少钱 | 每道菜估算成本，月度汇总 |
| 健身/饮食控制人群 | 复刻历史菜单 | 一键"再做一次"，按人数缩放 |

## 3. 核心价值主张

1. **拍照即所得** — 不输入文字、不选标签，一张图换一份完整食谱。
2. **AI 是辅助、用户是主** — AI 结果默认就保存，但任何字段都能就地编辑。
3. **从"记录"到"行动"** — 记录的菜可以一键加入购物清单、一键再做一次。
4. **看得见的成本** — 每道菜都有钱数，每周/月都有总账。

## 4. 用户旅程 (Golden Path)

### 4.1 首次启动
```
打开 → 登录/注册 → 3 屏引导卡片 → 立即拍一张 → AI 识别 → 保存第一条 → 首页有内容
```
目标：从冷启动到第一条记录 < 60 秒。

### 4.2 日常记录（核心循环）
```
首页 → 中间凸起按钮 → 拍照/选图 (选完立即开始识别)
  → 识别中 (三段式进度文案)
  → 结果页 (可就地编辑任意字段)
  → 保存 → 回首页
```
目标：拍照启动到保存完成 ≤ 4 次点击、≤ 30 秒。

### 4.3 复用
```
首页/搜索 → 点开一道菜 → "再做一次" → 调整人数 → 复制到今天
                       → "加入购物清单" → 自动合并食材
```

### 4.4 失败兜底
识别失败 / 用户不满意 → 三个出口：**换一张图** / **手动填写** / **反馈"识别不准"**（写入 ai_feedback，供 prompt 迭代）

## 5. 信息架构

底部 Tab，三个一级入口：

| Tab | 内容 | 默认页 |
|---|---|---|
| 日记 | 时间线 + 搜索 + 分类筛选 | 启动默认页 |
| 拍照 | 中间凸起按钮，唤起识别流程 | — |
| 我的 | 用户信息、本周/月成本面板、购物清单、个人价格表、设置、退出登录 | — |

**重要改动**：原 `home_screen` 右上角的"退出登录"必须迁到「我的」→ 设置；目前误点风险极高。

## 6. 功能列表（按版本）

### V1.0 — MVP+ (本期开发目标)

#### F1 AI 识别与就地编辑（P0）
- 选图后**立即触发识别**，不要"开始识别"按钮
- 结果页和编辑页**合并为一页**：默认漂亮排版，每个字段点击进入行内编辑
- 顶栏只有一个动作："保存到日记"
- 材料/步骤支持**增/删/改**：
  - 每条右侧编辑、删除图标
  - 列表末尾"+ 添加一项"
  - 步骤支持长按拖拽排序
- Tips 同样可增删

#### F2 识别中进度提示（P0）
三段式假进度（无需后端流式）：
- 0–2s：「识别食物中…」🍅
- 2–5s：「估算成本中…」💰
- 5s+：「生成制作步骤中…」📝

#### F3 识别失败的三个出口（P0）
错误页提供：换一张图 / 手动填写 / 反馈识别不准（多选 + 备注，POST `/ai/feedback`）。

#### F4 底部 Tab 导航 + 我的页（P1）
- 三个 Tab；中间为拍照凸起按钮
- 「我的」首屏：头像 + 昵称 + 本周成本卡片 + 入口列表（购物清单、价格表、设置、退出）

#### F5 首次引导 + 空状态强化（P1）
- 首次登录后 3 屏卡片（拍照即可记录 / AI 自动算成本 / 一键再做和买菜）
- 末屏 CTA："立即拍第一张"，直接唤起相机

#### F6 购物清单（P1，本期最重要的留存抓手）
- 食谱详情页：「加入购物清单」按钮
- 我的 → 购物清单页：
  - 已勾选 / 未勾选分组
  - 同名 + 同单位的食材自动合并数量
  - 长按可调整数量
  - "全部清空已勾选" 一键键
  - "分享给家人"（文本形式分享）

#### F7 再做一次（P1）
- 食谱详情页 FAB「再做一次」
- 弹窗确认人数（默认沿用原食谱）
- 按比例缩放材料数量和成本
- 复制为今天的新记录（cooked_at = today），原记录不动

#### F8 按食材搜索（P1）
- 首页搜索框增加 tab 切换：菜名 / 食材
- 后端 `GET /foods` 增加 `ingredient` 参数（详见 API.md）

#### F9 个人价格记忆（P1）
- 用户修改材料价格 → 写入 `user_ingredient_prices`
- 下次 AI 识别命中相同食材时，**用用户的最新价格覆盖** AI 估算
- 我的 → 我的价格表：可查看 + 手动调整 + 删除
- 列表展示：食材名 / 单价 / 单位 / 上次更新

#### F10 成本面板（P1）
- 我的页首屏卡片：本周烹饪支出 + 平均每餐成本
- 点开 → 月度详情页：
  - 月度总成本柱图（按周分组）
  - 最贵 Top 3 / 最便宜 Top 3
  - 同分类支出占比饼图

### V1.1 — 留存增强（V1.0 上线 2 周后）

- **F11 周报推送**：每周日 20:00 push「这周做了 X 道菜，估算省下 ¥Y 外卖费」
- **F12 复购检测**：保存时若同月内有同名菜或材料重合 ≥ 80%，提示「更新到已有记录？」
- **F13 分享卡片差异化**：菜谱长图 / 纯文本食谱 / 购物清单三种格式，每张带 App 水印
- **F14 离线草稿**：识别后没保存就退出，下次进入提示「上次有一份草稿未保存」

### V2.0 — 社区与营养（无明确日期）

- 美食地图（地理位置标签）
- 社交（关注 / 点赞 / 评论）
- AI 营养分析（热量、蛋白、脂肪、碳水）
- 多语言（英 / 日）
- 离线模式（本地缓存最近 100 条）

## 7. 关键交互细则

### 7.1 就地编辑约定
- 菜名：点击 → 输入框 + 软键盘 + 确认/取消按钮
- 单条材料：点击 → bottom sheet，包含 名称 / 数量 / 单位 / 单价（自动同步个人价格表）
- 单条步骤：点击 → bottom sheet，描述（多行文本）+ 耗时（数字）
- 删除：右侧 X 图标 + 一次确认；删除后顶部出现「撤销」snackbar

### 7.2 购物清单合并算法
- key = (规范化名称, 单位)；同 key → 数量相加
- 不同单位（如「番茄 2 个」vs「番茄酱 1 瓶」）保留为两条
- 名称规范化：去空格、繁简归一、忽略大小写
- 用户手动添加的条目优先级高于自动加入的

### 7.3 个人价格覆盖
- AI 返回的 `estimated_price` 为 fallback
- 后端在 `/ai/recognize` 响应前查 `user_ingredient_prices`：命中则用个人价格替换，并在响应里标记 `price_source: "user"`
- 客户端 UI 上对用户校准过的价格显示一个小角标

### 7.4 退出登录确认
两步确认（弹窗 → 确认按钮），避免误退导致草稿丢失。

## 8. 度量与北极星

**北极星**：周活用户人均记录数 ≥ 3

| 指标 | 当前 | V1.0 目标 |
|---|---|---|
| 拍照 → 保存漏斗完成率 | 未度量 | ≥ 70% |
| 编辑率（用户改了 AI 至少一个字段） | 未度量 | ≥ 30% |
| AI 识别 P95 耗时 | 未度量 | < 8s |
| AI 识别准确率（用户未做大范围修改） | 未度量 | > 85% |
| 次日留存 | 未度量 | ≥ 40% |
| 购物清单功能 7 日渗透率 | N/A | ≥ 25% |
| 再做一次月活渗透率 | N/A | ≥ 15% |

## 9. 埋点事件清单（详见 API.md `/events`）

| 事件名 | 触发时机 | 关键属性 |
|---|---|---|
| `app_open` | App 启动 | first_open: bool |
| `onboarding_step` | 引导卡片切换 | step: 1–3 |
| `camera_open` | 进入拍照页 | source: home/tab/empty |
| `image_picked` | 选完图 | source: camera/gallery |
| `recognize_start` | 调用 /ai/recognize | image_size: int |
| `recognize_done` | 识别返回成功 | latency_ms, cache_hit |
| `recognize_fail` | 识别失败 | error_code |
| `recipe_edit` | 用户改了任一字段 | field: name/ingredient/step/... |
| `food_save` | 保存成功 | from: recognize/manual |
| `food_view` | 打开详情页 | food_id |
| `food_share` | 点了分享 | format: card/text/shopping |
| `food_duplicate` | 再做一次 | scale_serving: bool |
| `shopping_add` | 加入购物清单 | food_id, ingredient_count |
| `shopping_check` | 勾选已购 | item_id |
| `ai_feedback` | 用户反馈识别不准 | reasons: list, food_id |

埋点客户端采用批量上报：每 30s 或 50 条触发一次 POST `/events`。

## 10. 非目标（V1.0 明确不做）

- 多人协作 / 家庭共享菜单
- 视频步骤
- 智能体对话式修改（"把这道菜改成低盐版"）
- 自建食材库 / 厨具库
- 营养成分

这些放在 V2 评估，不为它们让 V1 加复杂度。

## 11. 风险与依赖

| 风险 | 应对 |
|---|---|
| OpenAI 价格波动 / 限流 | Redis 缓存 + 失败兜底为手动模式 |
| 用户上传非食物图 | Prompt 增加"非食物则返回 unrecognized"指令，前端给出友好提示 |
| 个人价格表脏数据 | 单条价格如偏离市场均价 ±5× 则不参与覆盖，仅展示 |
| 识别准确率不达标 | ai_feedback 表收集 bad case，每周离线分析 prompt 调整 |
