# Delicious 产品需求文档 (PRD)

> 版本: v3 (2026-06-30 修订)
> 状态: V1.0 开发完成进入测试；V1.1 购物建议已编码完成；V1.2+ 规划中

## 1. 产品定位

一款 AI 驱动的"美食日记 + 厨房助手" App。用户拍一张菜的照片，AI 自动识别菜名、生成材料清单、估算成本、写出制作步骤。记录沉淀为个人菜单，可随时复用、生成购物清单、查看烹饪开销。

**一句话**：拍照即所得的美食日记，让"我做过/我想再做"的菜不再丢失。

## 2. 目标用户与场景

### 2.1 核心 Persona

不抽象描画，每个 persona 都有名字、月活假设和最频繁的场景。

**P1 — 林婷，28 岁，互联网设计师**
- 北京独居，周末偶尔做饭犒劳自己；爱在小红书发"美食笔记"
- 痛点：跟着小红书一边做一边看，结束就把菜谱忘了；下次想复刻无从下手
- 月活：8–12 道菜，集中在周末
- 价值锚点：拍完就有完整可复用的食谱，「再做一次」+「分享卡片」

**P2 — 陈昊，32 岁，刚搬出来住的程序员**
- 杭州，住公司附近一居室；之前外卖为主
- 痛点：想做饭省钱也想吃健康点，但「不知道一道菜要买什么、几钱、几步」
- 月活：4–6 道，工作日晚上为主
- 价值锚点：购物清单 +「看看去哪买」+ 成本估算

**P3 — 张妈妈，45 岁，全职太太**
- 上海，给三口之家做饭；月伙食 ¥2500
- 痛点：买菜花销心里没数；想攒一本「我们家的家常菜本」
- 月活：30+ 道，几乎每天
- 价值锚点：成本面板 + 个人价格表 + 复用历史菜谱

**P4 — 王哥，38 岁，健身教练**
- 深圳，自己 meal prep；高蛋白控碳水
- 痛点：复制相同菜单麻烦；按人数 / 餐数缩放材料每次都要算
- 月活：12–16 道，主要是周日一次性记录一周
- 价值锚点：「再做一次」按人数缩放 + 食材反查

### 2.2 反 Persona（V1.x 不优化的人群）

明确不为这些场景设计，保持决策清爽：
- **重度社区用户**：想看大量他人菜谱、关注 KOL — 去下厨房 / 豆果
- **专业营养师 / 减脂记录**：精确到克的卡路里输入 — 去薄荷 / MyFitnessPal
- **不下厨房只是想看食谱学习**：去 YouTube / B 站视频

### 2.3 竞品对照

| 产品 | 核心模式 | 与我们的关键差异 |
|---|---|---|
| 下厨房 / 豆果美食 | 大型菜谱社区 + UGC | 需要主动搜，不沉淀「我做过的」；无 AI 拍照 |
| 薄荷健康 / MyFitnessPal | 文字 / 条码日记 + 营养 | 输入门槛高，无 AI 拍照；不输出可操作的食谱 |
| Tasty / Yummly（海外） | 食谱推荐 + 视频 | 不本土化；无成本估算；不解决「记录」 |
| Google Lens / 各 AI 识图 | 单点识别 | 一次性，没沉淀，无复用，无购物链路 |
| 微信「美食小程序」类 | 工具化轻量 | 无账号体系；不能跨设备 |
| **Delicious** | 拍照→完整食谱→沉淀→复用 / 买菜 | 端到端闭环；AI 当辅助而非主角；成本透明 |

**核心定位**：不是「另一个菜谱 App」，而是**家庭厨房的私人记账本 + 助手**。社区是 V3 才碰的话题。

## 3. 核心价值主张

1. **拍照即所得** — 不输入文字、不选标签，一张图换一份完整食谱。
2. **AI 是辅助、用户是主** — AI 结果默认就保存，但任何字段都能就地编辑。
3. **从"记录"到"行动"** — 记录的菜可以一键加入购物清单、一键再做一次。
4. **看得见的成本** — 每道菜都有钱数，每周/月都有总账。

## 4. 用户旅程 (Golden Path)

### 4.1 首次启动
```
打开 → 登录/注册 → 3 屏引导卡片 → 立即拍一张 → AI 识别 → 保存第一条 → 首页有内容
```
目标：从冷启动到第一条记录 < 60 秒。

### 4.2 日常记录（核心循环）
```
首页 → 中间凸起按钮 → 拍照/选图 (选完立即开始识别)
  → 识别中 (三段式进度文案)
  → 结果页 (可就地编辑任意字段)
  → 保存 → 回首页
```
目标：拍照启动到保存完成 ≤ 4 次点击、≤ 30 秒。

### 4.3 复用
```
首页/搜索 → 点开一道菜 → "再做一次" → 调整人数 → 复制到今天
                       → "加入购物清单" → 自动合并食材
```

### 4.4 失败兜底
识别失败 / 用户不满意 → 三个出口：**换一张图** / **手动填写** / **反馈"识别不准"**（写入 ai_feedback，供 prompt 迭代）

### 4.5 出门买菜（V1.1 引入）
```
购物清单 → 顶部「🧭 看看去哪买」
  → 询问定位授权（已授权则跳过）
  → 3 Tab：附近超市 / 配送到家 / 外卖买菜
  → AI 一句话推荐策略：「永辉买 7 项 + 叮咚补紫苏，约 ¥40 最省」
  → 点店铺：跳第三方导航/App，或回退到网页
```
设计原则：信息密度做满（一眼看完三个 Tab 就知道选哪家），让用户跳出之前决策已经完成。

## 5. 信息架构

底部 Tab，三个一级入口：

| Tab | 内容 | 默认页 |
|---|---|---|
| 日记 | 时间线 + 搜索 + 分类筛选 | 启动默认页 |
| 拍照 | 中间凸起按钮，唤起识别流程 | — |
| 我的 | 用户信息、本周/月成本面板、购物清单、个人价格表、设置、退出登录 | — |

**重要改动**：原 `home_screen` 右上角的"退出登录"必须迁到「我的」→ 设置；目前误点风险极高。

## 6. 功能列表（按版本）

### V1.0 — MVP+ (本期开发目标)

#### F1 AI 识别与就地编辑（P0）
- 选图后**立即触发识别**，不要"开始识别"按钮
- 结果页和编辑页**合并为一页**：默认漂亮排版，每个字段点击进入行内编辑
- 顶栏只有一个动作："保存到日记"
- 材料/步骤支持**增/删/改**：
  - 每条右侧编辑、删除图标
  - 列表末尾"+ 添加一项"
  - 步骤支持长按拖拽排序
- Tips 同样可增删

#### F2 识别中进度提示（P0）
三段式假进度（无需后端流式）：
- 0–2s：「识别食物中…」🍅
- 2–5s：「估算成本中…」💰
- 5s+：「生成制作步骤中…」📝

#### F3 识别失败的三个出口（P0）
错误页提供：换一张图 / 手动填写 / 反馈识别不准（多选 + 备注，POST `/ai/feedback`）。

#### F4 底部 Tab 导航 + 我的页（P1）
- 三个 Tab；中间为拍照凸起按钮
- 「我的」首屏：头像 + 昵称 + 本周成本卡片 + 入口列表（购物清单、价格表、设置、退出）

#### F5 首次引导 + 空状态强化（P1）
- 首次登录后 3 屏卡片（拍照即可记录 / AI 自动算成本 / 一键再做和买菜）
- 末屏 CTA："立即拍第一张"，直接唤起相机

#### F6 购物清单（P1，本期最重要的留存抓手）
- 食谱详情页：「加入购物清单」按钮
- 我的 → 购物清单页：
  - 已勾选 / 未勾选分组
  - 同名 + 同单位的食材自动合并数量
  - 长按可调整数量
  - "全部清空已勾选" 一键键
  - "分享给家人"（文本形式分享）

#### F7 再做一次（P1）
- 食谱详情页 FAB「再做一次」
- 弹窗确认人数（默认沿用原食谱）
- 按比例缩放材料数量和成本
- 复制为今天的新记录（cooked_at = today），原记录不动

#### F8 按食材搜索（P1）
- 首页搜索框增加 tab 切换：菜名 / 食材
- 后端 `GET /foods` 增加 `ingredient` 参数（详见 API.md）

#### F9 个人价格记忆（P1）
- 用户修改材料价格 → 写入 `user_ingredient_prices`
- 下次 AI 识别命中相同食材时，**用用户的最新价格覆盖** AI 估算
- 我的 → 我的价格表：可查看 + 手动调整 + 删除
- 列表展示：食材名 / 单价 / 单位 / 上次更新

#### F10 成本面板（P1）
- 我的页首屏卡片：本周烹饪支出 + 平均每餐成本
- 点开 → 月度详情页：
  - 月度总成本柱图（按周分组）
  - 最贵 Top 3 / 最便宜 Top 3
  - 同分类支出占比饼图

#### F11 复购检测（P1）
- 保存时若同月内有同名菜或材料重合 ≥ 85%，后端返回 200 + duplicate_of
- 客户端弹窗三选项：更新原记录 / 新建一条 / 取消
- 详见 [API.md POST /foods] 与 [ARCHITECTURE §4.3]

### V1.1 — 留存与变现（V1.0 上线后 2~4 周）

#### F12 购物建议「看看去哪买」（P0，V1.1 头号优先级）

让用户从"列清单"走到"真的买到"。AI 的核心价值不是"番茄哪里有"，而是"这一整张清单去哪儿买最高效"。

- **入口**：购物清单顶部新增按钮「🧭 看看去哪买」（清单非空时显示）
- **位置授权**：首次点击弹系统授权；拒绝则回退到"按城市选择"
- **三个 Tab**：
  - **附近超市**：高德 POI 按距离排序，标注覆盖度（X/N 项）和缺哪几样
  - **配送到家**：盒马 / 叮咚 / 朴朴等同城配送 App
  - **外卖买菜**：美团买菜 / 京东到家
- **AI 推荐策略**：基于清单 + 用户位置 + 可达店铺，生成一句话最优组合
- **跳转**：点店铺 → 高德/百度地图导航；点 App → deeplink，失败回退到 web fallback
- **AI 该做与不该做**：
  - ✅ 食材语义归类（番茄=西红柿）、店铺类型对清单的覆盖度判断、一句话推荐文案
  - ❌ 找门店（用高德 API）、抓商品价格（V1.1 不做，反爬严重）、做比价
- **隐私**：坐标精度 100 米脱敏存储，仅用于本次推荐；用户可一键关闭定位
- 详见 [ARCHITECTURE §5 购物建议系统] 与 [API.md POST /shopping/buy-suggestions]

#### F13 周报推送（P1）
每周日 20:00 push「这周做了 X 道菜，估算省下 ¥Y 外卖费」

#### F14 分享卡片差异化（P2）
菜谱长图 / 纯文本食谱 / 购物清单三种格式，每张带 App 水印

#### F15 离线草稿（P2）
识别后没保存就退出，下次进入提示「上次有一份草稿未保存」

#### F16 token 持久化恢复（P1，技术补丁）
启动时尝试用 secure storage 里的 token 直接进入登录态，避免每次启动都要重登

## 7. 关键交互细则

### 7.1 就地编辑约定
- 菜名：点击 → 输入框 + 软键盘 + 确认/取消按钮
- 单条材料：点击 → bottom sheet，包含 名称 / 数量 / 单位 / 单价（自动同步个人价格表）
- 单条步骤：点击 → bottom sheet，描述（多行文本）+ 耗时（数字）
- 删除：右侧 X 图标 + 一次确认；删除后顶部出现「撤销」snackbar

### 7.2 购物清单合并算法
- key = (规范化名称, 单位)；同 key → 数量相加
- 不同单位（如「番茄 2 个」vs「番茄酱 1 瓶」）保留为两条
- 名称规范化：去空格、繁简归一、忽略大小写
- 用户手动添加的条目优先级高于自动加入的

### 7.3 个人价格覆盖
- AI 返回的 `estimated_price` 为 fallback
- 后端在 `/ai/recognize` 响应前查 `user_ingredient_prices`：命中则用个人价格替换，并在响应里标记 `price_source: "user"`
- 客户端 UI 上对用户校准过的价格显示一个小角标

### 7.4 退出登录确认
两步确认（弹窗 → 确认按钮），避免误退导致草稿丢失。

### 7.5 购物建议交互细则（V1.1）
- **位置授权时机**：仅在用户点了「看看去哪买」之后请求，不在 App 启动时弹（避免劝退）
- **店铺排序**：附近超市按 `score = 0.6 * 覆盖率 + 0.4 * (1 - 距离/最大距离)`，覆盖优先于距离
- **缺货降级**：超市覆盖不到 60% 时，AI 文案会主动建议"补 X 项可走叮咚"，把用户引到第二个 Tab
- **deeplink 兜底**：每条线上链接同时附带 web URL，跳第三方 App 失败时打开浏览器
- **空数据兜底**：当前定位附近 1 公里无 POI → 显示"换个地方试试"+ 手动输入城市的入口
- **不打扰**：用户已经选过店铺一次，30 分钟内再次点开默认折叠 AI 推荐文案，直接展示上次选择置顶

## 8. 度量与北极星

## 10. 非目标

**V1.0 不做**
- 多人协作 / 家庭共享菜单
- 视频步骤
- 智能体对话式修改（"把这道菜改成低盐版"）
- 自建食材库 / 厨具库
- 营养成分

**V1.1 购物建议明确不做**
- 抓取电商页面比价（反爬严重、数据不稳定，会被骂死）
- 用户评价聚合
- App 内承接交易（没那么多资源，跳第三方就行）
- 海外用户（高德覆盖国内，海外用 Google Places 等 V2 再说）

这些放在 V2 评估，不为它们让当期版本加复杂度。

## 11. 风险与依赖

| 风险 | 应对 |
|---|---|
| OpenAI 价格波动 / 限流 | Redis 缓存 + 失败兜底为手动模式 |
| 用户上传非食物图 | Prompt 增加"非食物则返回 unrecognized"指令，前端给出友好提示 |
| 个人价格表脏数据 | 单条价格如偏离市场均价 ±5× 则不参与覆盖，仅展示 |
| 识别准确率不达标 | ai_feedback 表收集 bad case，每周离线分析 prompt 调整 |
| **(V1.1)** 高德 POI API 5000 次/日额度 | 4 层缓存（食材别名 30 天 / POI 6h / 覆盖度 24h / 文案 4h），目标命中率 ≥ 90% |
| **(V1.1)** 用户拒绝定位授权 | 回退到"手动选城市 + 行政区"，仍能给出 POI（精度降低） |
| **(V1.1)** deeplink 不稳定（盒马/美团版本差异） | 每条跳转附带 web URL；记录跳转成功率，季度审视 |
| **(V1.1)** 跳出第三方 App 后不回流 | 入口处把决策辅助做到位（覆盖度 + 一句话推荐），让用户跳出前已决策完成 |
| **(V1.1)** 用户隐私（定位敏感） | 坐标精度脱敏到 100 米存储；仅在请求购物建议时使用；用户可一键关闭 |

## 13. 商业化路径

V1 / V2 阶段，**变现要让位于增长**。任何商业化动作必须满足：不打断核心流程、不强迫付费、关掉后不影响功能。

### 13.1 短期（V1.0–V1.2）— 不变现

理由：MAU < 10 万时变现单位价值低、信任成本高。把全部精力放在拉新和留存。

### 13.2 中期（V1.3 起）— CPS 联盟分佣

**「看看去哪买」是天然的入口**。基础设施已就位（`purchase_clicks` 表）。

- 接入对象：盒马、叮咚、京东到家、美团买菜
- 商业模式：点击 / 下单分佣（CPS），不动产品形态
- 必要条件：MAU ≥ 5 万、月均跳转 ≥ 10 万次（联盟谈判门槛）
- 不动作：不把分佣高的店强排前面，排序逻辑不变（覆盖优先于距离），否则毁信任

### 13.3 长期（V2 起）— 订阅会员

**Pro 用户**（建议 ¥18/月 或 ¥128/年）权益：
- AI 识别无限次（免费用户每月 30 次）
- 高级成本面板（年度报告、家庭对比）
- 家庭菜单共享（多账号同步）
- 营养分析 / 减脂模式
- 无广告（如果广告引入）

不做的：
- ❌ 解锁基础功能后才能用（违背产品理念）
- ❌ 内购虚拟商品（不是游戏）
- ❌ 内置广告位（影响信任）

### 13.4 度量与红线

| 指标 | 红线 |
|---|---|
| 付费转化率（Pro） | 目标 3% / DAU；< 1.5% 视为方案失败需要重做 |
| 购物建议跳转点击的 CPS 单点收益 | 国内同类目均价 ¥0.5~¥3 |
| 商业化对核心漏斗的负面影响 | 漏斗完成率绝对值下降 > 5% → 立刻回滚 |

## 14. 冷启动策略

V1.0 公测 4 周内冲到日活 1000 的最低可行计划。

### 14.1 用户来源（按权重）

| 渠道 | 投入 | 预期周拉新 | 关键动作 |
|---|---|---|---|
| 小红书 KOC 投放 | 高 | 200–400 | 找 5K~5W 粉的家庭烹饪 KOC，"用了 30 天 / 我的家常菜本"内容 |
| 朋友圈分享卡片裂变 | 中（产品内置） | 100–200 | 分享卡片必带二维码 + 邀请文案；扫码即下载 |

## 15. 隐私与合规

中国市场的合规线是**红线**，违反代价远高于功能价值。

### 15.1 法规依据

- 《个人信息保护法》(PIPL, 2021)
- 《网络安全法》(CSL, 2017)
- 《数据安全法》(DSL, 2021)
- 《App 收集使用个人信息违法违规行为认定方法》(2019)

### 15.2 数据分级与处理

| 数据类型 | 敏感级 | 收集时机 | 存储方式 | 保留期限 |
|---|---|---|---|---|
| 邮箱 / 昵称 / 密码 | 中 | 注册 | bcrypt(密码)；明文(邮箱/昵称) | 账号删除时清除 |
| 拍摄照片 | 中 | 用户主动拍 | R2 / MinIO，路径含 nonce | 永久（除非用户删除） |
| 食谱 / 备注 / 价格 | 低 | 用户主动保存 | PG | 同上 |
| 位置 (V1.1) | **高** | 用户点"看看去哪买"时按需 | events 表存 100m 脱敏；缓存 key 用 geohash5(~5km) | events 90 天；缓存 4h |
| 跳转点击 (V1.1) | 低 | 用户点击外链 | `purchase_clicks` | 1 年（分佣对账需要） |
| 埋点 | 低 | 持续 | `events`，按月分区 | 6 个月 |

### 15.3 必须实现的合规动作

| # | 动作 | 版本 |
|---|---|---|
| 1 | 首次启动展示《隐私政策》《用户协议》，必须主动勾选才能进入 | V1.0 |
| 2 | 注册前请求最小必要权限；位置 / 相机 / 相册按需弹（不在启动时） | V1.0 |
| 3 | 「我的」→「关于」展示完整隐私政策；可重读 | V1.0 |
| 4 | 「我的」→「账号与隐私」提供"删除我的账号"入口，48 小时内清除一切数据 | V1.1 |
| 5 | 「我的」→「账号与隐私」提供"导出我的数据"入口（JSON 下载） | V1.2 |
| 6 | 13 周岁以下用户保护：注册询问年龄，未成年禁止 | V1.0 |
| 7 | 第三方 SDK 清单公示（OpenAI、高德、boto3、bcrypt 等） | V1.0 |

### 15.4 数据出境

- OpenAI 调用涉及数据出境（图片 + 食材文字）
- V1 阶段以"不可避免的技术服务"列入用户协议
- V2 评估切换到国内大模型（智谱 / 百川 / Qwen-VL），同等准确率优先

## 16. 持续改进机制

### 16.1 用户反馈环

```
用户反馈 (App 内 / 微信群)
   ↓
分类（功能建议 / Bug / AI bad case / 体验问题）
   ↓
每周一过周会：取 Top 5 进 backlog
   ↓
1 周内必回复用户：感谢 + 修复时间预估
```

App 内反馈入口在「我的」→「意见反馈」（V1.2 加）。

### 16.2 AI bad case 处理

`ai_feedback` 表是金矿。每周固定动作：
- 周一上午：导出上周 feedback，归类（菜名错 / 材料漏 / 步骤怪 / 价格偏）
- 周三：根据高频问题更新 prompt，A/B 验证
- 周五：发本周 AI 准确率报告（自动跑 100 张图，对比上周）

目标：AI 准确率每月提升 ≥ 1 个百分点。

### 16.3 数据驱动决策（数据周会）

每周三下午 30 分钟看板：
- 漏斗：拍照启动 → 识别成功 → 编辑 → 保存
- 留存：D1 / D7 / D30
- 功能渗透：购物清单 / 再做一次 / 买建议
- 商业化（V1.3 起）：跳转点击数、CPS 收入
- 异常：识别失败率、API P99、客诉数

任何指标连续 2 周不达标 → 进入"待修复"列表，下次必谈。

### 16.4 A/B 测试框架（V1.3 起）

V1.0 / V1.1 不做 A/B，量太小没统计意义。V1.3 起对关键改动（如新增功能入口位置、AI 文案 prompt）走 A/B：
- 灰度 10% / 90% 起步
- 双指标：核心漏斗 + 该功能渗透率
- 必须明确"什么情况下停止实验"

### 16.5 用户访谈

每月 5 个用户深度访谈：
- 3 个高活用户（每周 ≥ 3 次打开）
- 1 个流失用户（30 天未打开）
- 1 个新用户（注册 ≤ 7 天）

记录访谈纪要到 `docs/research/`（V1.2 起建立）。

## 17. 产品决策记录 (PDR)

记录关键决策的**原因**，避免未来重新争论。**新决策追加，不删旧的。**

### PDR-001 不做社区 / UGC
- **时间**：2026-06-30 PRD v2
- **决策**：V1/V2 不做菜谱推荐流、不做关注/点赞/评论
- **原因**：社区是另一种产品形态（要做内容审核、推荐算法、运营）。我们的核心价值是"个人记账本"，社区会稀释这一点。下厨房 / 豆果已经把社区做得很重，绕开
- **何时复议**：MAU 突破 50 万；或用户访谈中明确表达"想看别人的菜谱"超过 30%

### PDR-002 AI 是辅助、用户是主
- **时间**：2026-06-30 PRD v2
- **决策**：AI 结果默认保存到日记，但每个字段都能就地编辑；不强迫用户审核
- **原因**：强迫审核会破坏"拍照即所得"的体验承诺。编辑的成本比从零开始低得多
- **后果**：编辑率指标 ≥ 30% 即视为成功（说明 AI 有价值但用户有控制权）

### PDR-003 购物建议不做比价
- **时间**：2026-06-30 PRD v2.1
- **决策**：V1.1 仅做 POI 推荐 + 覆盖度 + 跳转，不抓商品价格、不做平台比价
- **原因**：电商网页反爬严重；价格数据不稳定会被骂；接 API 需主体资质和谈判
- **何时复议**：MAU 突破 10 万，可以走联盟 API（如京东开放平台）拿到稳定价格流

### PDR-004 V1.0 砍掉营养分析
- **时间**：2026-06-30 PRD v2
- **决策**：MVP 不做卡路里 / 蛋白 / 碳水识别
- **原因**：营养识别需要重度数据库 + 用户精确测量，门槛远高于"识别菜名"。MVP 阶段加它会让产品变重
- **何时复议**：V2，作为 Pro 会员的差异化功能

### PDR-005 选 OpenAI，不自训
- **时间**：2026-06-30 ARCHITECTURE v2
- **决策**：使用 OpenAI GPT-4o Vision，不自建模型
- **原因**：自训需大量标注数据 + GPU 算力 + 团队；MVP 阶段验证产品价值更重要
- **何时复议**：DAU > 10 万 / OpenAI 单月费用 > ¥50,000 / 国产模型质量达到对等

### PDR-006 选高德，不选腾讯/百度
- **时间**：2026-06-30 ARCHITECTURE v2.1
- **决策**：购物建议 V1.1 POI 用高德 Web API
- **原因**：免费 5000 次/日额度最大；导航 deeplink (`uri.amap.com`) 体验最好；商家收录最全
- **何时复议**：高德调价 / 限流；或腾讯地图 POI 突破 10W 商家时

### PDR-007 复购检测用 200 + duplicate_of，不用 409
- **时间**：2026-06-30 API v2
- **决策**：相似记录返回 HTTP 200 + 数据字段（`duplicate_of`），不返回 409 Conflict
- **原因**：409 在 HTTP 语义上太硬；客户端体验上是"提示用户选"而非"操作失败"
- **争议**：违反 REST 语义纯洁性，但产品体验优先
- **何时复议**：API 给第三方开放时（需要更标准的语义）

### PDR-008 位置精度脱敏 100m
- **时间**：2026-06-30 ARCHITECTURE v2.1
- **决策**：events 表存位置时降到小数点后 3 位（~110m）
- **原因**：100m 精度足以做地区聚合分析；同时避免精确轨迹复原
- **何时复议**：PIPL 监管细则更新 / 用户投诉精度问题
